# Google Health / Fitbit Sync

Trail Runner Coach 2.1.0 adds direct cloud sync through the **Google Health API**. This is the preferred path for Fitbit and the Google wearable ecosystem because it uses Google OAuth and a server-side API rather than the legacy Google Fit REST flow.

## What the integration imports

The first release requests read-only access and normalizes the following data into the same schema already used by Apple Health, Strava, Garmin, Suunto and manual imports:

- Exercise/workout sessions
- Sleep duration
- Daily resting heart rate
- Daily heart-rate variability
- Steps
- Active energy
- Active minutes
- Walking/running distance
- Weight
- Body-fat percentage

Availability depends on the records present in the connected Google Health account, the source device/app, and the scopes the user grants. Missing data types are reported as warnings and do not block other readable data types.

## Architecture

```text
Fitbit / Pixel Watch / compatible Google Health source
                         ↓
                    Google Health
                         ↓ OAuth 2.0
        Cloudflare Wearable Sync Worker
                         ↓ encrypted token boundary
            normalized provider payload
                         ↓
      Cross-provider activity deduplication
                         ↓
        IndexedDB → Strain / Recovery / Readiness
```

Client ID, Client Secret and refresh tokens are never placed in `public/`. Provider tokens are encrypted before Cloudflare KV storage. The browser identifies itself to the Worker with a high-entropy device token whose SHA-256 hash is used server-side.

## One-time setup

### 1. Prepare Google Cloud

1. Create or select a Google Cloud project.
2. Enable **Google Health API**.
3. Configure the OAuth consent screen.
4. During testing, add your Google account as a test user when required.
5. Create an OAuth client of type **Web application**.
6. Keep the Client ID and Client Secret outside GitHub.

The exact Authorized redirect URI is generated by the setup command after the Worker has been deployed.

### 2. Run the setup wizard

From the repository root:

```bash
npm ci
npm run setup:google-health
```

The wizard:

- reuses or creates the three shared wearable KV namespaces;
- preserves existing Strava configuration;
- stores Google OAuth credentials as Cloudflare Worker secrets;
- deploys the wearable-sync Worker;
- prints the Authorized redirect URI;
- writes a secret-free `google-health-setup-result.json` receipt.

The generated receipt contains the Worker URL and callback metadata only. It does not contain the OAuth Client Secret, token-encryption key or user tokens.

### 3. Register the redirect URI

Return to the Google Cloud OAuth client and add the exact URI printed by the wizard, for example:

```text
https://trail-runner-coach-wearable-sync.<account>.workers.dev/oauth/google_health/callback
```

The scheme, host, path and trailing slash must match exactly.

### 4. Import and connect in Trail Runner Coach

1. Open **Log → Connections** / **บันทึก → เชื่อมต่อ**.
2. Open **Google Health / Fitbit Setup**.
3. Import `google-health-setup-result.json`.
4. Run the readiness check.
5. Select **Connect Google Health**.
6. Sign in with the Google account that contains the health/Fitbit data.
7. Review and grant the requested read-only scopes.
8. Return to the app and select **Sync**.

After the first successful connection, the existing Auto Sync and Retry Queue manage subsequent refreshes while the PWA is open or returns online.

## Requested scopes

The integration requests only read access needed by the current scoring model:

```text
openid
email
profile
https://www.googleapis.com/auth/googlehealth.activity_and_fitness.readonly
https://www.googleapis.com/auth/googlehealth.health_metrics_and_measurements.readonly
https://www.googleapis.com/auth/googlehealth.sleep.readonly
```

Location, ECG, nutrition and write scopes are not requested in 2.1.0.

## Deduplication policy

A Fitbit workout may also appear in Strava or another provider. Trail Runner Coach does not blindly insert every copy.

1. Provider + external activity ID is checked first.
2. Cross-provider matching then compares start time, duration, distance, activity type and average heart rate.
3. High-confidence matches become one canonical activity with multiple `sources[]` and `externalRefs[]`.
4. Uncertain matches are retained separately for user review.
5. Strain uses the canonical activity only once.

Daily metrics such as sleep, RHR, HRV and steps remain source-aware and preserve manually entered pain/fatigue fields.

## Operational limitations

- Live end-to-end authorization requires your Google Cloud OAuth credentials and cannot be completed from a deploy package alone.
- OAuth apps in testing mode may be restricted to configured test users and may require re-authorization depending on Google Cloud policy.
- A connected account may not contain every supported data type.
- The current implementation performs API sync when the app is open, focused or comes back online. A fully closed PWA cannot continuously execute browser-side imports.
- Google Health webhook subscriptions are not enabled in 2.1.0. They require project-level subscriber/subscription setup and signed-event verification, so this release intentionally uses authenticated API sync rather than an unverified public webhook endpoint.
- The Worker accepts partial data-type success: one unavailable endpoint does not discard valid exercise, sleep or body data from other endpoints.
- Before a public multi-user launch, complete Google app verification, privacy-policy and account-authentication requirements applicable to the requested health scopes.

## Troubleshooting

### `redirect_uri_mismatch`

The Authorized redirect URI in Google Cloud does not exactly match the Worker callback. Copy the URI shown in the in-app wizard or setup receipt.

### No refresh token

Disconnect, revoke the app grant if necessary, then connect again. The Worker requests offline access and explicit consent so Google can issue a refresh token.

### Connected but no data

Check:

- the chosen Google account is the account receiving Fitbit/Google Health data;
- the requested scopes were granted;
- the account has recent records for the selected date range;
- the Provider card warnings for unavailable data types;
- Activity Integrity for a workout merged with Strava.

### One workout appears twice

Open **Data & Wearables → Activity Integrity** and review the uncertain match. Future imports remember a Merge or Keep Separate decision.
